Database Security

Bad actors can come in any place where a user can interact with data through your website. Consequently, there are several layers of security necessary.

Front end protections are limited, due to users' ability to manipulate the DOM. HTML attributes and labels are malleable and cannot be trusted to make it to the back end.

Data entered can also take the form of HTML or SQL code designed to trick the back end into reporting information different from what is intended. These attacks are referred to as injection attacks. Fortunately, such inputs can be sanitized prior to use in a database capacity.

Whichever tech stack is used, data submitted by the user should be secured as much as possible on the front- and back-ends of the website. Securing the data takes the form of a three-part approach.

  • Validate

    Ensure the data is consistent with whats is being asked for in the form. Things like ages or quantities should be integers or floats. Dates should be formatted consistent with the final storage format. Strings should not contain any special code, or, if necessary, all code will need to be rendered inert at a later stage.

  • Sanitize

    Now that the code has been validated, it should be sanitized with any potential code replaced with inert text. Input should never be fed directly to the database as part of a query, or served up by the web-server as potential HTML.
